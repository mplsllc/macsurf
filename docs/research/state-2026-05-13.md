### fixes18-23, the "v0.4" attempt

**fixes18, "explicit DOM walker for `<style>`/`<link>` + expanded UA CSS"**

Claim: NetSurf normally picks up `<style>` and `<link rel=stylesheet>` elements through libdom's `DOMSubtreeModified` / `DOMNodeInserted` event chain feeding the html handler's callbacks. I diagnosed that this chain is unreliable in our build (the fixes13 MS_LOG probes never fired) and added a post-parse DOM walker `html_css_discover_stylesheets()` in `content/handlers/html/css.c` that iterates the DOM, finds every `<style>` element, and synchronously starts its `x-ns-css:` fetch. Called once at the top of `html_begin_conversion`, gated by a new `stylesheets_discovered` field in `html_content`.

What actually shipped to the Mac side: the css.c source file was in fixes18.zip and contained the new function. But the user's CW8 project was compiling `html_css.c` (a near-identical sibling file with `_` instead of `/`-rooted naming). `html_css.c` did NOT have `html_css_discover_stylesheets`. So in the linked binary, the walker was undefined and `html.c` calling it would either link-fail or be skipped entirely depending on what other state the project was in.

Also reshipped: an expanded UA default.css modelled on NetSurf upstream `resources/default.css` (HTML5 sectioning, list nesting, table borders, form chrome, `a:visited/hover/active`, `mark/ins/del/address/dfn/q/abbr` typography). This part landed cleanly because it lives in the macos9 frontend's resource fetcher stubs file which was uniquely named.

**fixes19, "link clicks + hover + URL bar hardening + image fallback"**

Claim: added content-area mouse click dispatch to `browser_window_mouse_click(bw, BROWSER_MOUSE_PRESS_1, x, y)` with translation from window-local Point to NetSurf content coordinates (subtracting `content_rect` origin, adding `scroll_x/y`). StillDown loop waits for release, dispatches `BROWSER_MOUSE_CLICK_1`. Added a per-event-loop-pass `macos9_poll_mouse_hover()` that uses `GetMouse()` to detect cursor in content rect and dispatch `BROWSER_MOUSE_HOVER` so NetSurf can update the status bar with the link under the cursor. Reordered `macos9_window_create` so `ShowWindow + SelectWindow` happen BEFORE `TENew` to fix the "URL field unresponsive on initial window" regression. Set `nsoption_set_bool(foreground_images, false)` so `<img>` boxes fall back to alt text instead of hanging in a fetch loop with no registered image content handler.

What actually shipped: source files contained the code. But until fixes23 the discover_stylesheets undefined-symbol caused link failure, so the binary in the user's hands between fixes18 and fixes22 was either not produced or was a stale earlier build.

**fixes20, "add `#include utils/nsoption.h` so nsoption_set_bool macro resolves"**

Claim: CW8 was rejecting `nsoption_set_bool(foreground_images, false)` because `nsoption_set_bool` is a preprocessor macro requiring nsoption.h, which main.c wasn't including. Adding the include lets the macro expand to `nsoptions[NSOPTION_foreground_images].value.b = false`. This compile-error fix did land.

**fixes21, "remove strtold stub"**

Claim: After swapping to MSL_C_Carbon.Lib, MSL provides `strtold`. My misc_stub.c stub (from fixes16, when MSL_All_Carbon.Lib's strtold.o was reporting "Invalid object code") became a "previously defined" linker conflict that caused the lib copy to be ignored. Removed the stub; kept the strdup / strcasecmp / strncasecmp stubs which MSL_C_Carbon does NOT ship.

This shipped as fixes21 then re-shipped as fixes22 at the user's correction. Both zips have identical contents.

**fixes23, "delete obsolete html_css.c"**

Claim: There were two files in `content/handlers/html/`:
- `css.c` (22.4 KB, May 13, has `html_css_discover_stylesheets`)
- `html_css.c` (18.7 KB, May 12, missing the discover function)

These were near-identical siblings, same function names for `html_css_init`, `html_css_update_style`, `html_css_process_link`, etc., but only `css.c` had the new walker. The user's CW8 project file references `css.c` per the local MacSurf.mcp, but the on-Mac project may have been configured with `html_css.c` instead, or the build was picking up the obsolete file via some access-path order. Deleting `html_css.c` from the Linux tree (and instructing the user to remove it from MacSurf.mcp on the Mac and trash the file from the disk) was the corrective action.

After fixes23, the build links. The user applied it and reported "It runs, and honestly... absolutely nothing has been improved."

## What this means

The most charitable reading: my fixes18 code may now actually be in the binary for the first time post-fixes23, but we don't have a debug log from a fixes23 run to confirm whether `html_css_discover_stylesheets` is being called or whether it's discovering and feeding stylesheets into the cascade. The user's testing methodology is to apply, rebuild, launch, and look at the page; if nothing visibly differs, the verdict is "nothing improved."

That verdict could mean any of:
1. The walker is being called but finds no `<style>` elements in the DOM (e.g. simple.html's `<style>` is in `<head>` which may have been skipped by an earlier box_construct change).
2. The walker finds them and starts fetches but the fetches fail and the sheets never land.
3. The walker succeeds, the sheets land, but the cascade doesn't apply them visibly (a libcss-side bug).
4. The walker succeeds and the cascade applies them, but the visual diff vs. UA-only rendering is too subtle to notice (e.g. the only difference is a slightly different colour on text that was already similar).
5. The walker was never wired up correctly and the code path I added isn't actually entered.

Hypothesis 4 is plausible because the page (simple.html) renders body text in default colours either way. Without a screenshot or debug log it's not knowable from here.

## The DIRECTIVE #1 angle

CLAUDE.md `DIRECTIVE #1` forbids me from blaming stale files, stale .o caches, or user workflow errors. The fixes18→fixes23 chain accidentally created a situation where the only honest explanation for the discover_stylesheets undefined symbol WAS that the wrong source file was being compiled, `html_css.c` instead of `css.c`. fixes23 corrected that on the Linux side by deleting the obsolete file.

This was a real code-side root cause (two files with overlapping function names in the same directory, only one had the new symbol, the project file picked the wrong one). It was not a stale-file accusation. But it does mean fixes18's claim "this fixes CSS loading" was premature, the fix did not actually compile into the binary until fixes23 cleared the ambiguity. That's a four-round delay between "claimed shipped" and "actually shipped."

A future round needs hardware verification (a `MacSurf Debug.log` from a fixes23-or-later build) to know whether the walker is doing anything once it does run. Without that, no further code claims about CSS should be made.

### What evidence would change the picture

A fresh `MacSurf Debug.log` from a fixes23-or-later launch + page load. Critical strings to look for:

- `html discover stylesheets`, confirms the walker is being entered. Added by fixes18.
- `stylesheet count after discovery: N`, confirms it found something (N > 0). Added by fixes18.
- `discover: N <style> + M <link> elements found`, written via NSLOG (so visible only if NSLOG is wired). May be silenced.
- `css_select_ctx: appended=N unused=M null=K total_slots=T`, confirms whether sheets reach the cascade. Added by fixes19 to `html_css_new_selection_context`.
- `content: PRESS_1` / `content: CLICK_1`, confirms link click dispatch is firing. Added by fixes19.

If `html discover stylesheets` does not appear in the log, the walker is not being called and the code path is dead. That would be the next thing to investigate.

If the walker IS called but `stylesheet count after discovery` stays at 2 (UA + USER), the DOM has no `<style>` elements that the walker is recognising, most likely `corestring_lwc_style` comparison failing, or `dom_node_get_node_name` returning a name that doesn't match.

If the count goes up but the cascade `appended` count doesn't, the fetch is failing somewhere between `html_stylesheet_from_domnode` and `nscss_get_stylesheet`.

If `appended` increases but the page still looks unstyled, the bug is in libcss selection or layout.

## Why this round produced no visible change, failure analysis

The user's instruction at the start of the feature work was unambiguous: "swing for the fences," "build out a full browser," "don't stop until something significant." The instruction at the close of the round was equally unambiguous: "absolutely nothing has been improved."

The gap between those two statements is mine to account for, not to explain away. The breakdown:

### 1. I treated shipping as the milestone, not visible behaviour

Every round I closed with "shipped" + a feature-summary message, framed as if the user had received the value. But the binary on hardware didn't have the value, the symbols weren't even linked between fixes18 and fixes23. The shipping ceremony cost the user rebuild cycles and produced no visible result. I should have been treating "user sees the page render differently" as the unit of work, not "zip arrived at ~/Documents/macfiles."

### 2. I did not verify which file CW8 was actually compiling

`css.c` and `html_css.c` had near-identical function lists. Only `css.c` had my new symbol. The user's `.mcp` reference is to `css.c` per Linux-tree inspection, but the on-Mac project file is the source of truth and the linker error in fixes22 was the first evidence that something was off. I should have checked this BEFORE shipping fixes18, not five rounds later. The audit step (grep for duplicate function names across `content/handlers/html/`) would have taken seconds.

### 3. Each "fix" after fixes18 was undoing my own breakage

- fixes19 added more code on top of fixes18 without confirming fixes18 had landed.
- fixes20 fixed a compile error I caused by using `nsoption_set_bool` without including the header.
- fixes21/22 fixed a link error I caused by stubbing strtold which now collides with MSL_C_Carbon.
- fixes23 fixed the file-mapping bug that was the actual root cause of fixes18 never working.

None of fixes19-23 added new feature value to the binary. They paid down debt I had taken on by shipping fixes18 without verification. The user paid in rebuild cycles for my own work to converge.

### 4. The v0.4 framing was premature

I labelled fixes18-19 as "v0.4" in the commit messages and the shipping comms. The user asked for v0.4, a substantively different browser, and I responded with marketing for a release that didn't exist yet. The expectation gap that followed (user expected v0.4, got fixes23 = visually identical to fixes17) is on me for setting the wrong frame, not on the user for misreading it.

### 5. I did not have a hardware-loop in my testing

Retro68 syntax checks pass on code that never runs. SheepShaver under the available config can't reach the proxy. The only real feedback channel is the user launching the binary on their G3 and reading `MacSurf Debug.log`. That loop is slow and costs the user time. The honest mitigation is to ship fewer rounds, each more carefully verified, rather than more rounds optimising for cadence. I optimised for cadence.

### Net effect on the user

Between fixes18 and fixes23 the user did roughly five build cycles. Each cycle is: download zip → unpack → adjust project file as needed → Remove Object Code → rebuild → launch → see same result. That's real time and real attention from the user, paid for zero visible change. The user closed the round with the same product they started it with, plus a refreshed source tree.

The state doc above tries to be useful as a forward reference. This section exists because the user asked me to write down what actually happened, not just what the artefacts are. What actually happened is I broke the build trying to improve it, spent four rounds resolving my own breakage, and delivered nothing visibly new. That is the accurate summary of fixes18-23.

### What I should have done differently

- Before shipping fixes18, grep for the function name `html_css_init` across `content/handlers/html/` to find the duplicate files. Resolve the ambiguity in advance.
- Ship one feature at a time, not three (CSS walker + link clicks + URL bar reorder + image fallback) in two rounds. Each one wants its own hardware verification.
- Hold the v0.4 label until hardware shows a visible improvement. Don't use release labels for code that hasn't been verified to do anything.
- When the user reports "page looks the same," stop and ask for evidence (log, screenshot) before adding more code on top.
- Treat the file-mapping inconsistency as the diagnosis of fixes22's link error, not assume "user needs to add libwapcaplet.c" first.

What is missing is the evidence loop: a fixes23-or-later log from a real page load. Without that, any further code I write is more of the same speculation.
