## Category 1, Missing headers

Now narrowed to four. All four exist on the Linux build host. All four
are referenced by both the failing files AND files that compile cleanly
in the same directory.

### The smoking gun: `azimuth.c` vs `autogenerated_align_content.c`

```c
/* browser/libcss/src/parse/properties/azimuth.c — COMPILES CLEAN */
#include <assert.h>
#include <string.h>

#include "bytecode/bytecode.h"
#include "bytecode/opcodes.h"
#include "parse/properties/properties.h"
#include "parse/properties/utils.h"
```

```c
/* browser/libcss/src/parse/properties/autogenerated_align_content.c — FAILS */
#include "bytecode/bytecode.h"
#include "bytecode/opcodes.h"
#include "parse/properties/properties.h"
#include "parse/properties/utils.h"
```

Same four `#include` lines. Same parent directory on Linux
(`browser/libcss/src/parse/properties/`). Both files appear in
`MacSurf.mcp` as `<PATH>../../../../libcss/src/parse/properties/<name>.c</PATH>`
under identical `<PATHTYPE>Project</PATHTYPE>` entries. CW8 happily
compiles `azimuth.c`, `outline.c`, `border_spacing.c`, `clip.c` etc. and
silently fails on every `autogenerated_*.c` it tries.

The only structural difference between the two sets of files is **how
they got onto the Mac**:

- `azimuth.c` and friends were already on the Mac as part of the v0.1.0
  libcss source-tree drop. They sit in
  `Macintosh HD:…:browser:libcss:src:parse:properties:` and CW8 has been
  building them for months.
- The `autogenerated_*.c` files were created **after v0.1.0** as part
  of the libcss port that split the upstream single `autogenerated.c`
  into one file per CSS property. They were delivered to the Mac inside
  one of the v0.2 zips. Their physical location on the Mac is whatever
  the user's unpack script put them at.

### Root cause hypothesis

CW8's `#include "header.h"` resolution searches in this order:

1. The directory containing the **source file currently being
   compiled** (the "user-include" rule).
2. Each `UserSearchPath` entry in project order.

If the `autogenerated_*.c` files were physically extracted into a
**different folder** than `browser:libcss:src:parse:properties:` on
the Mac (e.g. the user's `~/Documents/macfiles/` staging area, or a
flat top-level folder, or a v0.2-specific subfolder), then:

- Step 1 looks in that staging folder for `bytecode/bytecode.h` ,
  obviously not there.
- Step 2 should fall back to the `UserSearchPath` entries, including
  `{Project}/../../../../libcss/src` which would resolve correctly.
- BUT, CW8 has a long-documented quirk where `UserSearchPath` entries
  with multi-level `..` segments behave differently depending on the
  source file's actual on-disk location vs. the path declared in the
  project's FILELIST. Specifically, if the project's FILELIST `<PATH>`
  doesn't physically point at where the file lives (e.g. via a Finder
  alias, a symlink, or because the file lives elsewhere and CW8 found
  it via project resolution rather than filesystem resolution), the
  search-path expansion can use the FILELIST `<PATH>` as the base
  rather than the file's actual parent, and the relative search
  paths land somewhere unintended.

The other files in the same folder (`azimuth.c` etc.) were placed
correctly during the original v0.1 unpack and CW8's Step 1 finds
`browser:libcss:src:bytecode:bytecode.h` directly **without ever
consulting the user search paths**.

This explains *exactly* why the same `#include` line works in one
file and fails in another in the same project, the working files
never need the search path; the failing files do, and the search path
is broken (or pointing somewhere unexpected) on the Mac.

### Fix, investigation, not yet a code change

Three things to check on the Mac, in order:

1. **Locate the `autogenerated_*.c` files on the Mac filesystem.**
   In Finder, navigate to the project folder. Browse to
   `browser:libcss:src:parse:properties:`. Confirm whether
   `autogenerated_align_content.c` is physically in that folder
   alongside `azimuth.c`. If it is **not**, find where it actually
   lives, most likely in the unpack staging dir.

2. **If the files are misplaced**, move them into
   `browser:libcss:src:parse:properties:` so they sit beside
   `azimuth.c`. CW8's user-include rule will then resolve
   `bytecode/bytecode.h` the same way it does for azimuth.

3. **If the files are already in the right folder**, then CW8 IS
   reaching the search-path fallback and the `{Project}/../../../../libcss/src`
   entry isn't resolving on the Mac. In that case the fix is to
   add an **absolute Mac-style path** as a new User Path entry in the
   IDE: Project Settings → Access Paths → User Paths →
   `Macintosh HD:…:browser:libcss:src:`. The Phase 3/4 `content/handlers`
   path proved that absolute (or correctly-anchored) paths work; the
   `..`-relative form may simply not be reliable past 3 levels.

## Proposed fix order

Highest dependency first.

1. **[blocks all 1,747 errors]** Verify the physical on-disk location
   of the `autogenerated_*.c` files on the Mac. Use Finder; do NOT rely
   on the .mcp file alone since the failure modes diverge from what
   the .mcp claims. Acceptance: a Get Info on
   `autogenerated_align_content.c` shows it living in
   `…:browser:libcss:src:parse:properties:`, the same folder as
   `azimuth.c`.

2. **[if step 1 finds them misplaced]** Move them. The fix is a Finder
   move, no source change, no MacSurf.mcp change. Re-run the build.

3. **[if step 1 finds them in the right folder]** Add an absolute
   Mac-style search path for libcss/src in the CW8 IDE:
   `Macintosh HD:…:browser:libcss:src:` as a new User Path. This is
   the same workaround proposed in attempt 1's report for the four-`..`
   relative path issue, narrowed now to just the libcss/src case
   (since content/handlers proved the three-`..` form works).

4. **[regardless of which fix worked]** Document the resolution in
   CLAUDE.md so the next clean checkout has the lesson, either
   "v0.2 delivery zips must preserve the libcss/src/parse/properties/
   tree structure" or "libcss internal headers require an absolute
   User Path in the CW8 IDE".
