## Root cause #1, libdom internal header tree missing

**Same pattern as the libcss-headers fix**: libdom's `.c` files are in
the project, the `.h` files were never delivered to the Mac. 30 unique
`cannot be opened` headers in this batch:

```
core/attr.h               core/cdatasection.h        core/comment.h
core/doc_fragment.h       core/document.h            core/document_type.h
core/element.h            core/entity_ref.h          core/namednodemap.h
core/node.h               core/pi.h                  core/string.h
core/text.h               events/mutation_event.h    html/html_button_element.h
html/html_collection.h    html/html_document.h       html/html_element.h
html/html_input_element.h html/html_select_element.h html/html_table_element.h
html/html_tablecaption_element.h                     html/html_tablerow_element.h
html/html_tablesection_element.h                     html/html_text_area_element.h
treebuilder/element-type.h    utils/validate.h
parser.h                  dom/bindings/hubbub/parser.h
dom/bindings/hubbub/errors.h
```

All exist on the Linux side under `browser/libdom/{src,include,bindings}/`.
None exceed the HFS+ 31-char limit.

**The 94 `illegal use of incomplete struct 'dom_node_internal'`
errors and the ~12 `dom_html_document` / `dom_parser` undefineds are
all downstream of this**, once the dom node header doesn't open,
every `node->base.vtable = …` line becomes "illegal use of
incomplete struct" because CW8 only knows the forward declaration.

The `_dom_*_initialise(...) redeclared as int (...)` errors are
particularly diagnostic: they're the C89 implicit-int rule firing
because the function's prototype header (e.g.
`html/html_table_element.h`) didn't open, so CW8 saw the call site
first with no return type, defaulted to `int`, then later met the
real definition and complained about the mismatch.

The `redraw.c:1006 continue;` and `libdom.c:56 break;` "illegal use
of keyword" errors are also almost certainly downstream, both
keywords are textually inside valid `for` / `switch` constructs but
CW8's parser state has been corrupted by hundreds of upstream
syntax errors and lost track of what scope it's in.

### Fix

Ship a `macsurf-libdom-headers.zip` containing every `.h` under
`browser/libdom/`. 204 files, well-formed, none over 31 chars,
small total size. Deliver to laptop, unpack on Mac. The unpack
will create new folders for any subdirs the Mac doesn't have yet
(probably `core/`, `events/`, `html/`, `treebuilder/`, `bindings/`).

This fix alone will clear approximately **2,800 of the 3,088
errors** based on the cascade structure.

## Root cause #2, `autogenerated_propset.h` has C99 mixed declarations

This is a **real source-level issue we missed in the libcss audit**.
File: `browser/libcss/src/select/autogenerated_propset.h`. Around
lines 919, 955, and at least 27 other sites, probably one per
property setter macro:

```c
/* line 916-921 of autogenerated_propset.h */
*bits = (*bits & ~COUNTER_INCREMENT_MASK) | (((uint32_t)type & 0x1) <<
                COUNTER_INCREMENT_SHIFT);                       /* statement */

css_computed_counter *old_counter_arr = style->counter_increment;  /* DECL after statement */
css_computed_counter *c;                                            /* DECL after statement */
```

C89 forbids declarations after statements within a block. The fix
is to hoist every offending declaration to the top of its enclosing
block. CW8 then accepts it.

Symptom in errors.txt: 27 × `undefined identifier 'old_string_arr'`
and 18 × `undefined identifier 'old_counter_arr'`, those names exist
in the file but CW8 silently dropped them after the syntax error and
later references see them as undefined.

### Why the audit missed it

The libcss audit grepped for designated initializers, `inline`, `//`,
for-scope decls, etc., but **not for "declaration after statement"**,
which is harder to grep mechanically (you need to track block state).
Add this to the port-audit checklist for future libraries.

### Fix

Open `browser/libcss/src/select/autogenerated_propset.h` and find
every block where a declaration follows a statement. Hoist the
declaration to the top of that block. The file is auto-generated
upstream so this is a one-time port edit, not a per-rebuild fix.

29 lines mention `old_counter_arr` / `old_string_arr` in the file
grep suggests roughly 6-8 distinct functions need the hoist.

## Root cause #3, `struct jsthread *jsthread` field name collision

File: `browser/netsurf/content/handlers/html/private.h:152`.

```c
struct jsthread *jsthread;
```

CW8 reports `illegal name overloading`. C89 nominally allows a
field to share its name with a struct tag (different namespaces),
but CW8 specifically rejects it when both names appear in the same
declaration. Three call sites trigger it (probably whichever .c
files include `html/private.h`).

### Fix

Rename the field. Two reasonable options:

```c
struct jsthread *js_thread;        /* simple disambiguation */
struct jsthread *jsthread_ctx;     /* slightly more descriptive */
```

Plus updating the ~5-10 callers that read `c->jsthread`. NetSurf
core JavaScript is gated out via `WITHOUT_DUKTAPE` in our prefix
file, so this field is **never actually used** at runtime, it's
purely a header-shape issue. The cheapest fix is to rename it to
`js_thread`.

### `va_copy` redefined (1×)

Macro `va_copy` defined twice. Likely once by MSL stdarg.h and once
by our prefix or libcss/talloc. Fix: `#ifndef va_copy` guard around
the redefinition. Single-line cleanup, low priority.

### `internal compiler error in CError.c line 861` (1×)

CW8 bug. Triggers when error count exceeds some internal limit and
the error formatter overflows. Will go away on its own once the
real errors are fixed.

## Proposed fix order

Strict dependency order, each step reduces the error count enough
to make the next step's errors readable.

1. **[blocks ~2,800 errors]** Build and ship
   `macsurf-libdom-headers.zip` containing every `.h` under
   `browser/libdom/`. Deliver via the standard SCP path. User
   unpacks on the Mac, which will create new `core/`, `events/`,
   `html/`, `treebuilder/`, `bindings/` subfolders under
   `browser:libdom:src:` and `browser:libdom:include:dom:`. **Do this
   first**, almost everything else is invisible until it lands.

2. **[blocks ~150 errors]** Patch
   `browser/libcss/src/select/autogenerated_propset.h` to hoist
   every declaration that follows a statement. Mechanical edit, ~6-8
   functions. Re-run cross-check. Commit.

3. **[blocks ~30 errors]** Rename the `jsthread` field in
   `content/handlers/html/private.h:152` to `js_thread`, update the
   handful of `c->jsthread` callers in the html/ tree. One-line per
   site. Commit.

4. **[< 5 errors expected to remain]** Re-run the full build. Whatever
   surfaces will be much more interesting:
   - The `plot_style_bdr` undefined, if it survives steps 1-3, it's
     a real bug worth investigating.
   - The `va_copy` redefinition, quick `#ifndef` guard.
   - Anything new that the cascade was hiding.

5. **[forecast, not blocking]** Add "declarations after statements"
   to the port-audit checklist in CLAUDE.md so the next library port
   audit catches it.

## Forecast for attempt 4

After steps 1-3, the most likely remaining issues are:

- **Multi-arg `NSLOG` parse errors** (Phase 4 punt). Hundreds of
  upstream NSLOG calls pass 4+ args; the prefix file's NSLOG macro
  is 3-arg only. CW8 may have been silently dropping the extras
  until now, or it may error loudly once parser state is clean.
- **`%zu` printf in `layout_flex.c:722, 914`** (Phase 4 punt). MSL
  printf doesn't support `%zu`. Compile-time format strings aren't
  validated by CW8 so this won't be a build error, it'll be a
  runtime garbage-output bug surfaced only by inspecting the log.
- **More incidental cascades** in libdom or libhubbub if those
  libraries also have C99 mixed-decl idioms our audit missed.
