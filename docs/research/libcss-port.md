# libcss Port Audit (CW8 / C89)

## Status

**Research only — no code changes.** This document is a complete inventory of
what is in the libcss source tree and exactly what stands between it and a
clean CodeWarrior 8 / strict C89 / Mac OS 9 build. Same structure as the
parserutils, libhubbub, and libdom audits.

## Where the source lives

`browser/libcss/` — present in the repo, vendored alongside the other
NetSurf libraries. Not a submodule. License is **MIT**.

Upstream is the NetSurf project's libcss:
`https://git.netsurf-browser.org/libcss.git/` (or the GitHub mirror at
`github.com/netsurf-browser/libcss`).

**Nothing needs to be fetched.** The source is already where we need it.

## C99 / CW8 incompatibilities

### 1. `inline` keyword — 28 files

The largest occurrence so far. Spread across `parse/`, `select/`,
`utils/`, public headers, and several `select/properties/*.c` files.

Same fix as the prior libraries — `#define inline` is already in
`macsurf_prefix.h`. **No prefix change needed.**

### 2. `//` line comments — **REAL, first time at scale**

**1 file with start-of-line line comments:**
- `src/parse/mq.c` — lines 686-688, 975-976. Look like commented-out
  pseudo-code or grammar notes.

**14 files with trailing line comments** (`; // comment` form):
- `src/parse/properties/utils.c`
- `src/select/properties/font_size.c`
- `src/select/properties/flex_shrink.c`
- `src/select/properties/flex_basis.c`
- `src/select/properties/line_height.c`
- `src/select/properties/order.c`
- `src/select/properties/pitch.c`
- `src/select/properties/volume.c`
- `src/select/properties/column_count.c`
- `src/select/properties/speech_rate.c`
- (4 more — full sweep produces ~14 unique files)

**This is the first library where `//` is real and not just MIT URL
header noise.** Each one needs converting to `/* ... */`.

**Fix shape:** sed pass with the right regex:
- For trailing `; // comment` → `; /* comment */`
- For start-of-line `//` → `/* */`

The risk with sed is multi-line `//` comments — but if every `//` is a
single-line trailing remark or a single-line block, sed handles it
mechanically.

### 3. C99 designated initializers — **REAL, 3 files**

```
src/parse/mq.c                       lines 1200-1208 (event handler struct)
src/select/format_list_style.c       lines 484-510 (list-style structures, multiple)
src/select/unit.c                    lines 376, 405 (font_size structures)
```

`mq.c` is the worst — initializes a nested `event_handler` struct. The
others are simpler `name`, `system`, `symbols` field initializers.

**Fix shape:** same as libdom — convert to assignment-statement form.
Estimated 30-40 lines edited across the 3 files.

### 4. C99 for-scope declarations — **REAL, first time, 4 files**

```
src/parse/properties/utils.c:436     for (int i = 0; i < 4; i++)
src/select/computed.c:265            for (size_t i = 0; i < CSS_N_PROPERTIES; i++)
src/select/select.c:1039             for (size_t i = 0; i < CSS_ORIGIN_AUTHOR; i++)
src/select/select.c:1040             for (size_t j = 0; j < CSS_PSEUDO_ELEMENT_COUNT; j++)
src/select/properties/helpers.h:87   for (size_t i = 0; i < count; i++)
src/select/properties/helpers.h:121  (same)
src/select/properties/helpers.h:157  (same)
```

Total: 7 instances across 4 files. CW8 strict C89 will reject all of
them.

**Fix shape:** mechanical conversion. Each `for (TYPE i = 0; ...)`
becomes:
```c
TYPE i;
for (i = 0; ...; ...) { ... }
```
Note that `helpers.h` instances may be inside `static inline` functions
that get included into multiple translation units — the `i` declaration
needs to be at the start of the enclosing block in each case.

## Build-time codegen — **the hardest part of this port**

libcss has **two** codegen paths. Both must run on Linux before the
generated outputs can be checked in.

### Path A: select/* generators (already done — outputs in tree)

`src/select/select_generator.py` and `src/select/assets.py` are Python
scripts that produce:
- `src/select/autogenerated_computed.h`
- `src/select/autogenerated_destroy.inc`
- `src/select/autogenerated_propset.h`
- `src/select/autogenerated_propget.h`

**These files ARE already committed to the tree.** Verified by `ls`. No
action needed unless we ever modify the generator inputs.

### Path B: parse/properties/* — **the new hard part**

`src/parse/properties/css_property_parser_gen.c` is a host C program
(reads stdin args, writes a `.c` file). The Makefile does:

```make
$(BUILDDIR)/gen_parser: $(DIR)css_property_parser_gen.c
    $(BUILD_CC) -o $@ $^

$(DIR)autogenerated_$1.c: $(DIR)properties.gen $(BUILDDIR)/gen_parser
    $(BUILDDIR)/gen_parser -o $@ '$(shell grep "^$1:" $(DIR)properties.gen)'
```

So the build steps are:

1. Compile `css_property_parser_gen.c` with Linux gcc to produce a
   `gen_parser` host binary.
2. For each property line in `properties.gen` (119 of them), invoke
   `gen_parser -o autogenerated_<property>.c '<grep'd line>'` to produce
   one `.c` file per property.
3. Force-add all 119 generated files to the repo (the upstream
   `.gitignore` does NOT explicitly exclude them, but they're build
   artifacts so we're treating them the same way as parserutils'
   `aliases.inc` and libhubbub's `entities.inc`).

**Build-system implication:** the most elaborate codegen we've hit. But
self-contained — no external tools beyond a C compiler and `grep`.

`properties.gen` has 119 property rules (verified via
`grep -cE '^[a-z][a-z_]+:' properties.gen`).

**All 119 file names ahead of time:** can be enumerated with the same
shell expression the Makefile uses (`perl -pe '...'` extracting the
property name from each line).

## Existing MacSurf shims that conflict

### `frontends/macos9/libcss/libcss.h` — **MAJOR conflict, biggest yet**

```
$ wc -l browser/netsurf/frontends/macos9/libcss/libcss.h
2013
```

**2,013 lines.** Even bigger than `dom/dom.h`'s 1,638. Same
monolithic-API-stub pattern as `dom/dom.h` — recreates large portions
of the libcss public API by hand.

**Reachability check:** `frontends/macos9/libcss/` is **not on any
current MacSurf user search path**. Same situation as `frontends/macos9/dom/`
was before the libdom port — orphaned but on disk. The 25-file grep we
did during the libdom port confirmed that the existing CW8 build
doesn't include any file that references libcss headers, so the stub
is not being used today.

**Fix shape:** delete `browser/netsurf/frontends/macos9/libcss/libcss.h`
and the now-empty `libcss/` directory as part of the wiring step. The
real libcss public headers take over via the new include path.

**This will be the largest single deletion in any port** (2,013 vs
1,638 for dom.h vs 37 for parserutils' utf8.h stub).

### `frontends/macos9/css/css.h` and `css/utils.h`

These are **NetSurf-core CSS handler stubs**, not libcss stubs. Same
situation as the `html/` stubs vs libhubbub during the libhubbub port.

```
frontends/macos9/css/css.h     24 lines    (NetSurf core css/css.h stub)
frontends/macos9/css/utils.h   115 lines   (NetSurf core css/utils.h stub)
```

These are headers in the NetSurf-core `css/` namespace, not the
`libcss/` namespace. They will conflict with `browser/netsurf/content/handlers/css/css.h`
and `browser/netsurf/content/handlers/css/utils.h` once those NetSurf-core
files are added to the build (later milestone). **They are not directly
on the libcss critical path.** Out of scope for this milestone.

(Same as `frontends/macos9/html/` was for the libhubbub port.)

### Wrong-named utils file in libcss

The libcss source has `src/utils/parserutilserror.c`. NetSurf and the
other ported libraries use a different convention for the same kind of
file (`parserutils_error_to_lib_error`-style helper). This is NOT a
conflict — just a slightly different file name. No action needed.

## What is required to ship this on the Mac

The porting work breaks down into **ten** concrete steps. This is the
biggest port yet by a wide margin.

1. **Build the host generator on Linux:**
   `gcc -o /tmp/libcss_gen_parser browser/libcss/src/parse/properties/css_property_parser_gen.c`
   This is the only port that requires building a host program from
   the library's own source.

2. **Generate 119 `autogenerated_*.c` files** by iterating over
   `properties.gen` and running the host generator for each property.
   Force-add all 119 outputs to the repo.

3. **Convert 7 for-scope `for` declarations** to C89 form in 4 files:
   - `src/parse/properties/utils.c:436`
   - `src/select/computed.c:265`
   - `src/select/select.c:1039, 1040`
   - `src/select/properties/helpers.h:87, 121, 157` (three sites — note
     this is a header that gets inlined into many translation units)

4. **Convert designated initializers to assignment-statement form** in
   3 files. The `mq.c` instance is the most complex (nested struct);
   the other two are straightforward field assignments. Estimated
   30-40 lines edited.

5. **Convert `//` comments to `/* */`** across ~15 files. sed pass on
   the trailing `; // ` form, manual review for any multi-line block.

6. **No `inline` work needed** — `#define inline` already in prefix.

7. **No `restrict` work needed** — `#define restrict` already in prefix.

8. **No new shims required.** `<strings.h>` and `<sys/types.h>` shims
   exist from prior ports.

9. **Delete `browser/netsurf/frontends/macos9/libcss/libcss.h`**
   (2,013 lines) and the now-empty parent dir. The real libcss public
   headers take over via the include path.

10. **Wire libcss into `MacSurf.mcp`:**
    - Add 2 user search paths: `{Project}/../../../../libcss/include`
      and `{Project}/../../../../libcss/src`
    - Add 302 `<FILE>` entries in tier order
    - Add 302 `<FILEREF>` entries in `LINKORDER`
    - **Exclude** `parse/properties/css_property_parser_gen.c` from the
      project — it's a host program, not a build target
    - Insert after the libdom block, before the NetSurf Core Utils block,
      so the link order resolves bottom-up
      (parserutils → hubbub → dom → css → netsurf core)

## What this audit does NOT cover

- Actually doing any of the ten steps above (this is research only).
- Verifying the generator output is valid C89 — once we run
  `gen_parser` for the first time we'll find out whether the generated
  files themselves use any C99 features. The generator was last touched
  in 2010 and may emit `inline` or `//`. That's a first-use
  discovery, not something this static audit can predict.
- Whether NetSurf core's `content/handlers/css/css.c` will need libcss
  symbols beyond the public `<libcss/libcss.h>` API. Same risk class
  as the libdom port had with NetSurf-core HTML handler.
- Whether there are libcss-internal dependencies between
  `select/properties/*.c` files that require ordering beyond the
  rough tiering above.
- The `propget.h`/`propset.h` autogenerated headers in `select/` are
  large files (each ~5K LOC). They're already in the tree but they may
  contain C99 features themselves (they're code-generator output).
  Worth a read-pass before the first build attempt.

## Bottom line

libcss is **the hardest port in the chain by every measure**:

- **302 `.c` files** to add to `MacSurf.mcp` — more than libparserutils
  + libhubbub + libdom combined (140).
- **119 build-time-generated files** that need a host program built
  first.
- **First library with real `//` line comments at scale** (~15 files).
- **First library with C99 for-scope declarations** (4 files, 7 sites).
- **Largest stub deletion yet** (2,013 lines vs 1,638 for dom.h).
- **Tied with libdom** for designated initializer fixes (3 files each).

But all the issues are **mechanical**, not architectural:

- Every C99 feature has a known fix shape from prior ports.
- The codegen is more elaborate than parserutils/hubbub but still
  self-contained — no external tools beyond a C compiler.
- Zero new shims required.
- No POSIX dependencies (beyond `<strings.h>` which is already shimmed).

**Hardest parts will be:**

1. **The CW8 project file edit** — 302 `<FILE>` and 302 `<FILEREF>`
   entries is the largest XML edit yet. Generation by Python loop is
   the only practical approach.
2. **The first run of the host generator** — first time we'll see what
   the generated files actually look like. They may need their own
   pass for C99 features.
3. **First-build link errors** when the real libcss headers replace the
   2,013-line stub. Anything that referenced the stub's slightly-different
   type signatures will need to migrate (no current build references it,
   but future NetSurf-core files will).
4. **The 7 for-scope conversions** in 4 files, particularly the 3 sites
   in `helpers.h` which is included into multiple .c files — each `i`
   declaration needs to be at the start of its enclosing block within
   the static inline helper, which may require small refactors.

Most likely first-build failure is **a C99 feature inside an
auto-generated file** that we can't predict from the source-tree audit
alone.
