# libhubbub Port Audit (CW8 / C89)

## Status

**Research only — no code changes.** This document is a complete inventory of
what is in the libhubbub source tree and exactly what stands between it and
a clean CodeWarrior 8 / strict C89 / Mac OS 9 build. Same structure as
`docs/research/parserutils-port.md`.

## Where the source lives

`browser/libhubbub/` — present in the repo, vendored alongside
`browser/netsurf/` and `browser/libparserutils/`. Not a submodule; the tree
is checked in directly. License is **MIT**.

Upstream is the NetSurf project's libhubbub:
`https://git.netsurf-browser.org/libhubbub.git/` (or the GitHub mirror at
`github.com/netsurf-browser/libhubbub`). The version in the tree is a
tagged release from the NetSurf side; the embedded README is the upstream
README verbatim.

**Nothing needs to be fetched.** The source is already where we need it.

- **No tests in scope** — `browser/libhubbub/test/` excluded.
- **No examples or perf in scope** — `examples/` and `perf/` excluded.

## C99 / CW8 incompatibilities

### 1. `inline` keyword — only 4 files

```
src/utils/parserutilserror.h         1 occurrence (static inline hubbub_error_from_parserutils_error)
src/treebuilder/in_cell.c            1 (close_cell)
src/treebuilder/in_table.c           2 (clear_stack_table_context, process_input_in_table)
src/tokeniser/tokeniser.c            5 (emit_character_token, emit_current_chars,
                                        emit_current_tag, emit_current_comment,
                                        emit_current_doctype)
```

Total: 9 `static inline` definitions across 4 files. Same fix as
libparserutils — `#define inline` is already in `macsurf_prefix.h` from
the parserutils port. **No prefix change needed; existing override
already covers libhubbub.**

### 2. `//` comments — none

Same false-positive situation as libparserutils. The audit script ran
`grep -E '^[[:space:]]*//'` and `grep -E ';[[:space:]]*//[^/]'` (start-of-line
line comments and trailing line comments) across all 49 files. **Both
returned zero matches.** Every `//` substring in the tree is the URL
`http://www.opensource.org/licenses/mit-license.php` inside a `/* */` MIT
license header block.

**Step is a no-op.** Same lesson learned as parserutils — `//` URLs in
license headers don't count.

### 3. `<stdint.h>` and `<stdbool.h>` — used pervasively

Every public header pulls in `<stdint.h>` and `<stdbool.h>`. The
`uint8_t`/`uint32_t`/`bool` types appear throughout. Same situation as
libparserutils — already shimmed for MacSurf via:

- `frontends/macos9/shims/stdint.h` — provides all the integer types
- `frontends/macos9/shims/inttypes.h` — now includes `stdint.h` (fixed
  during the parserutils port)
- The MacSurf prefix file's `_STDINT_H`/`_INTTYPES_H` predefines block
  MSL's C++ stdint chain
- `<MacTypes.h>` provides `enum { false, true }` for `bool`

**No new shim work required.** Same chain that handled libparserutils
will handle libhubbub.

### 4. `<inttypes.h>` — pulled in by 4+ files

Used by `parser.h`, `types.h`, `functypes.h`, `tree.h`, plus several
internal files. No `PRI*` format-macro usage in the source — only the
type definitions. Already covered by the existing
`shims/inttypes.h` → `shims/stdint.h` chain set up during the parserutils
port.

### 5. `<strings.h>` — POSIX, used by 2 files

```
src/treebuilder/treebuilder.c    #include <strings.h>
src/charset/detect.c             #include <strings.h>  (uses strncasecmp 5+ times)
```

**This is a new dependency that libparserutils did not have.** `<strings.h>`
is the POSIX header that declares `strcasecmp` and `strncasecmp`. CW8/MSL
does not provide it as a header; on the Mac, those functions live in
`<string.h>` (without the `s`).

**Existing infrastructure:** there is **no** `frontends/macos9/shims/strings.h`
today. This needs to be created. The fix is trivial — a one-line shim:

```c
#ifndef MACOS9_SHIMS_STRINGS_H
#define MACOS9_SHIMS_STRINGS_H
#include <string.h>   /* strcasecmp / strncasecmp live in string.h on CW8 */
#endif
```

(MSL's `string.h` does provide both functions — confirmed by the fact
that `frontends/macos9/macos9_fetch.c` already calls `strcasecmp` without
including `<strings.h>`.)

### 6. `entities.inc` — generated by Perl, not in tree

`src/tokeniser/entities.c` does:

```c
#include "entities.inc"
```

`entities.inc` is **not** in the tree. It is generated by
`build/make-entities.pl` from `build/Entities` (a 2,142-line table of
HTML5 named entities → Unicode codepoints). The Perl script builds a trie
representation and emits a static C array.

The `.gitignore` for libhubbub explicitly excludes `src/tokeniser/entities.inc`.

**Build-system implication:** Same pattern as libparserutils' `aliases.inc`.
Run `perl build/make-entities.pl` once on Linux, force-add the resulting
`src/tokeniser/entities.inc`, treat as a checked-in artifact.

### 7. `autogenerated-element-type.c` — generated by gperf, not in tree

This is **a second codegen step** that libparserutils did not have. It is
the bigger build-system risk in libhubbub.

`src/treebuilder/element-type.c` does:

```c
#include "treebuilder/element-type.h"

/* Auto-generated by `gperf`. */
#include "treebuilder/autogenerated-element-type.c"
```

`autogenerated-element-type.c` is generated from
`src/treebuilder/element-type.gperf` by the GNU `gperf` tool (perfect hash
function generator). The Makefile rule is:

```make
$(DIR)autogenerated-element-type.c: $(DIR)element-type.gperf
    gperf --output-file=$@.tmp $<
    sed -e 's/^\(const struct element_type_map\)/static \1/' $@.tmp >$@
    rm $@.tmp
```

So the chain is:

1. `gperf` reads `element-type.gperf` (a small DSL file listing all HTML
   element names + their `element_type` enum values).
2. `gperf` outputs an optimal-collision hash table of those names as C source.
3. A `sed` post-processing pass marks the `element_type_map` table `static`
   so it doesn't conflict at link time.

The `.gitignore` excludes
`src/treebuilder/autogenerated-element-type.c` for the same reason.

**`gperf` is NOT installed on this Linux machine** — verified via `which gperf`.
Two options:

- (a) `apt install gperf` (it's a tiny package, ~100 KB), then run the same
  Makefile rule on Linux, force-add the result.
- (b) Hand-write the equivalent table without gperf. The
  `element-type.gperf` source has under 100 entries; a simple linear
  scan or a hand-rolled `bsearch` would work and avoid the dependency.
  More code to maintain but fewer build dependencies.

Option (a) is the path of least resistance and matches what we did for
parserutils' `aliases.inc` — install the tool once, generate once, commit
the artifact.

## What is NOT a problem

I checked for and did not find:

| Feature | Result |
|---|---|
| `__VA_ARGS__` variadic macros | Not used anywhere |
| C99 designated initializers (`.field = ...`) | Not used |
| For-scope declarations (`for (int i = 0; ...)`) | Not used |
| `restrict` keyword | Not used |
| Compound literals (`(struct foo){...}`) | Not used |
| `long long` | Not used |
| Flexible array members (`int arr[];`) | Not used |
| Forward enum declarations | Not used |
| Variable-length arrays | Not used |
| `__attribute__` / `__builtin_*` | Not used |
| `snprintf` / `vsnprintf` | Not used |
| `%zu` printf format | Not used |
| `iconv` | Not used in libhubbub at all (parserutils has it; libhubbub does not) |
| `<errno.h>` | Not used |

This is **even cleaner** than libparserutils. libhubbub has no iconv
dependency at all (the README mentions iconv only because libparserutils
needs it, not libhubbub itself), and no debug-only `%zu` traps.

## Existing MacSurf shims that may conflict

### `frontends/macos9/html/html.h` and `html/form_internal.h`

These exist in the tree but are **NetSurf core HTML content handler stubs**,
not libhubbub stubs. They define `struct html_stylesheet`,
`struct content_html_object`, etc. — types used by NetSurf's
`content/handlers/html/html.c`, which is a layer above libhubbub.

**They do not directly conflict with libhubbub headers** because libhubbub's
public headers all live under the `hubbub/` namespace
(`<hubbub/parser.h>`, `<hubbub/tree.h>`, etc.) and never include anything
from `html/`. The two namespaces are independent.

These stubs will need attention when we wire NetSurf core's `html_init`
back in (a later milestone), but they are not on the libhubbub critical
path. Audit them at that point, not now.

### No `frontends/macos9/hubbub/` directory exists

There is no existing MacSurf stub for libhubbub itself. The real headers
will be visible from the start once the search path is added — no
conflict to resolve.

## What is required to ship this on the Mac

The porting work breaks down into eight concrete steps.

1. **Install gperf on Linux** (`apt install gperf`). One-time prerequisite
   for step 3.

2. **Generate `entities.inc`** by running
   `perl browser/libhubbub/build/make-entities.pl` from inside
   `browser/libhubbub/`. Force-commit the resulting
   `src/tokeniser/entities.inc` to the repo (it's gitignored).

3. **Generate `autogenerated-element-type.c`** by running gperf on
   `src/treebuilder/element-type.gperf` and applying the `sed` post-pass
   from the Makefile. Force-commit the result.

4. **Create `frontends/macos9/shims/strings.h`** as a one-line shim
   that includes `<string.h>`. (This is the only NEW shim required — the
   stdint/inttypes/stdbool chain from the parserutils port already covers
   everything else.)

5. **No `inline` work needed** — `#define inline` is already in
   `macsurf_prefix.h` from the parserutils port.

6. **No `//` comment work needed** — there are no real C++ line comments
   in the libhubbub tree.

7. **No stub deletion needed** — there is no existing
   `frontends/macos9/hubbub/` directory. The unrelated `frontends/macos9/html/`
   stubs are NetSurf-core territory, not libhubbub, and are out of scope
   for this milestone.

8. **Wire libhubbub into `MacSurf.mcp`:**
   - Add 2 user search paths: `{Project}/../../../../libhubbub/include`
     and `{Project}/../../../../libhubbub/src`
   - Add 30 `<FILE>` entries in tier order
   - Add 30 corresponding `<FILEREF>` entries in `LINKORDER`
   - Insert the new entries after the libparserutils block, before the
     NetSurf Core Utils block, so the link order resolves bottom-up
     (parserutils → hubbub → netsurf core)

## What this audit does NOT cover

Out of scope for this document, deferred to follow-ups:

- Actually doing any of the eight steps above (this is research only).
- Verifying that `apt install gperf` works on this Linux box and that the
  generated output matches what an upstream Hubbub release ships with.
- Whether hand-rolling `element-type` lookup (skipping gperf entirely) is
  worth the complexity to avoid the build dependency.
- Whether NetSurf core's `content/handlers/html/` will need libhubbub
  symbols beyond the public `<hubbub/parser.h>` API. We'll find out when
  we wire `html_init` later.
- Whether the existing `frontends/macos9/html/` stubs need to be retired
  or kept once the real NetSurf HTML content handler is built. Out of
  scope until the libhubbub→libcss→libdom chain is done.
- libhubbub's tree-construction callbacks need a real `hubbub_tree_handler`
  implementation to actually parse anything. That's frontend code, not
  library code, and it lives at the libdom layer (the next milestone).

## Bottom line

libhubbub is **another easy library**, slightly bigger than libparserutils
but cleaner overall. The audit checklist is shorter:

- **Zero** real C99 features beyond `inline` (already neutralized in our
  prefix file).
- **Zero** `//` line comments.
- **Zero** new POSIX dependencies except `<strings.h>` (one-line shim).
- **Zero** iconv usage.
- **Two** codegen artifacts to generate (`entities.inc` from Perl,
  `autogenerated-element-type.c` from gperf).

The hardest part — same as last time — will be **wiring the 30 files into
`MacSurf.mcp`** in the right order without breaking the existing build.

Most likely first-build failure is **the gperf install or its output**.
Second most likely is the search-path resolution in CW8 for the new
`{Project}/../../../../libhubbub/...` paths — same risk we have with
libparserutils that we won't actually verify until a real CW8 build attempt.
