# HFS+ Filename Truncation Mapping

## Background

HFS+ on Mac OS 9 enforces a hard 31-byte limit on file *names* (not paths).
CodeWarrior 8 reads source files via the Mac File Manager, so any source
file copied across with a basename >31 chars will be truncated by the
Finder during the unpack and CW8 will then fail to find it under the path
recorded in `MacSurf.mcp`.

The Mac OS X / Linux side of the toolchain has no such limit, so files
can be created on the build host with arbitrary names, this audit
catches them before they leave the host.

## Audit scope

Every file added or modified between `v0.1.0` (commit `7b17048`) and
`HEAD` was checked. Filenames with basenames > 31 characters:

| Directory | Count |
|---|---:|
| `browser/libcss/src/parse/properties/` | 29 |
| (everything else added/modified in v0.2) | 0 |

All 29 offenders share the same prefix: `autogenerated_*.c` files split
out from the upstream single-file `autogenerated.c` during the libcss
port. The `autogenerated_` prefix alone is 14 bytes, leaving only 17
bytes (including the `.c` extension) for the property name itself ,
not enough for the longer property names like `background_attachment`,
`border_bottom_color`, `list_style_position`, etc.

No collisions with the existing non-prefixed property files
(`background.c`, `border_color.c`, `border_style.c`, …), those use a
different naming scheme entirely.

## Truncation rule

Replace the leading `autogenerated_` (14 bytes) with `ag_` (3 bytes).
Saves 11 bytes per file, which is enough to bring every offender under
31 characters with margin to spare. The `ag_` prefix:

- Preserves uniqueness against the unprefixed `background.c`,
  `border_color.c`, etc. that already exist in the same directory.
- Stays mnemonic, clearly marks the files as machine-generated.
- Is short enough that even the longest property name
  (`background_attachment`, 21 chars) fits comfortably:
  `ag_background_attachment.c` = 26 bytes.

## Files NOT renamed

The 90 other `autogenerated_*.c` files in the same directory whose
basenames are already ≤31 bytes are kept under their original names.
Mixing `autogenerated_*` and `ag_*` is ugly but functional, and
renaming the short ones too would dirty the diff against upstream
without solving any actual problem. (If it later turns out the project
file maintenance is easier with a single naming scheme, the unmodified
files can be renamed in a follow-up pass.)

No files in any other directory needed renaming.

## Application checklist

When applying this rename:

1. `git mv` each file in the repo to its new name.
2. Update every `<PATH>` and `<FILEREF>` entry in
   `browser/netsurf/frontends/macos9/MacSurf.mcp` (each renamed file
   appears in three places: FILELIST `<FILE>`, LINKORDER `<FILEREF>`,
   GROUPLIST `<FILEREF>`).
3. There are no `#include "autogenerated_*.c"` lines anywhere (these
   files are independent translation units, not included), so no
   source patching is needed.
4. Re-run the cross-check on a few of the renamed files to confirm
   nothing in libcss expects the long name at the C level, none
   should, since the files are referenced only by the build system.
