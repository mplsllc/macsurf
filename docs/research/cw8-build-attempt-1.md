# CW8 Build Attempt 1, Error Analysis

## Source

`errors.txt` at repo root, 6,247 lines after CR→LF, **2,084 distinct
`Error :` lines**. The user noted this is a **batch, not the complete
list**, the failing files are alphabetically `a*` through `font_variant`
(37 source files), so a full build would emit considerably more from
the same root causes.

## TL;DR

**One root cause produces all 2,084 errors.** Five header `#include`s
fail at the very top of each translation unit; when the headers fail,
every type they define becomes unknown, and every subsequent function
declaration that uses those types collapses into "declaration syntax
error" / "identifier expected" / "illegal function definition" /
"undefined identifier 'error'" cascades.

Fix the five headers and the entire 2,000-error mountain disappears.
There are zero genuine type-mismatch, duplicate-symbol, or link-time
errors in this batch.

## Category 1, Missing headers (THE ONLY REAL CATEGORY)

Five distinct headers fail to open. All five exist on the Linux build
host at the path the search-path setting in `MacSurf.mcp` should
resolve to. The issue is therefore **not missing files**, it's
**search-path resolution on the Mac side**.

| Header | Resolved Linux path | Affected `#include` site | Used by |
|---|---|---|---|
| `bytecode/bytecode.h` | `browser/libcss/src/bytecode/bytecode.h` | libcss `parse/properties/*.c` line 16 | 36 libcss parse-property files |
| `bytecode/opcodes.h` | `browser/libcss/src/bytecode/opcodes.h` | libcss `parse/properties/*.c` line 17 | same 36 files |
| `parse/properties/properties.h` | `browser/libcss/src/parse/properties/properties.h` | libcss `parse/properties/*.c` line 18 | same 36 files |
| `parse/properties/utils.h` | `browser/libcss/src/parse/properties/utils.h` | libcss `parse/properties/*.c` line 19 | same 36 files |
| `html/font.h` | `browser/netsurf/content/handlers/html/font.h` | `content/handlers/html/font.c` line 29 | 1 file (this batch) |

### Root cause

`MacSurf.mcp` declares the relevant `UserSearchPath` entries:

```
{Project}/../../../../libcss/src                    (#12, libcss internal)
{Project}/../../../content/handlers                 (#12b, Phase 3+)
{Project}/../../../content/handlers/html            (#12c, Phase 4)
```

From `{Project}` = `browser/netsurf/frontends/macos9/`, these resolve
on Linux to `browser/libcss/src` and `browser/netsurf/content/handlers`
respectively, both of which exist and contain the missing headers. So
on Linux the cross-check passes, the failure is Mac-side only.

The plausible Mac-side causes, in decreasing likelihood:

1. **The Mac-side `MacSurf.mcp` is older than the repo HEAD.** The
   build was attempted on a project file that predates Phase 3/4.
   These three search-path entries (#12, #12b, #12c) didn't exist in
   the project file at v0.1.0; they were added during the wiring
   phases, but the Mac copy of `MacSurf.mcp` may not have been resynced
   since. Strong evidence: every error references `autogenerated_*`
   filenames, but the repo HEAD has just renamed 29 of those files to
   `ag_*`. The Mac side is on a pre-rename project file.

2. **CW8 path translation**, `MacSurf.mcp` uses Unix-style `..`
   relative paths. CW8 traditionally wants Mac-style colon-separated
   paths (`{Project}::..`). It typically accepts `..` too, but if the
   Mac unpacked the file fork into a folder layout that breaks the
   relative count, the four-`..` path would land outside `Macintosh
   HD:` entirely and silently resolve to nothing. The CLAUDE.md
   "Access Paths (CodeWarrior)" section already lists Mac-style paths
   as the canonical form for the manual setup, those are not the
   same paths as what the .mcp file currently encodes.

3. **Search path order or non-recursive flag**, CW8 search paths are
   non-recursive by default. If the entry was added but the
   recursive flag isn't set on `libcss/src`, `bytecode/bytecode.h`
   still resolves (one level deep) but if the entry got stored
   pointing one level too deep, the leaf `bytecode/` subfolder
   wouldn't be reachable.

### Fix

**Highest-leverage fix, order of investigation:**

1. **Re-sync `MacSurf.mcp` to the Mac.** The repo HEAD already
   contains the correct three search-path entries. The renames-zip
   shipped at the end of the v0.2 delivery includes the latest
   `MacSurf.mcp`. Confirm the Mac is using THAT version, not an
   older copy. If the Mac is on the v0.2 base zip, the renames-zip
   needs to be unpacked over the top.

2. **If the project file is current and the errors persist**, the
   issue is search-path resolution. Two sub-fixes to try in order:

   a. **Convert the four-`..` paths to absolute Mac-style paths** in
      the CW8 IDE: open Project Settings → Access Paths → User Paths,
      and replace each `{Project}/../../../../libcss/src` with the
      actual Mac path that points at the libcss source root, e.g.
      `Macintosh HD:patrick:macsurf-source Folder:browser:libcss:src:`.
      Same for `content/handlers` and `content/handlers/html`. Save.
      This is what the existing CLAUDE.md "Access Paths" section
      already documents for the previously-known-good paths, extend
      it with three new entries.

   b. **If absolute paths work**, regenerate `MacSurf.mcp` so that
      future syncs don't reintroduce the broken `..` paths. Either
      hand-edit the `<VALUE>` entries to absolute Mac paths, or write
      a tiny generator that emits the correct form. This should be a
      docs/CLAUDE.md update at minimum.

## Category 2, Missing symbols (link time)

**None in this batch.** Every error in `errors.txt` is a compile-time
error from the C front end. The build never reached the link stage ,
five `### Error: Compilation aborted at end of file ###` markers
confirm the compiler bailed before any TU was successfully translated.

## Category 3, Type mismatches

**None directly.** All `undefined identifier 'error'`, `'css_fixed'`,
`'unit'`, `'length'`, `'pctx'`, `'num'`, `'consumed'` are downstream
of the missing headers. With `int32_t` and the libcss `css_*`
typedefs unknown, every variable declaration that uses them turns
into a "declaration syntax error" and the compiler then sees the
identifier alone in the next statement and reports it as undefined.

`uint32_t unit = 0;` → "expression syntax error" at the same level ,
again, only because the missing libcss `bytecode/bytecode.h` would
have brought in `int32_t` / `uint32_t` indirectly via libparserutils.
With nothing pulling those typedefs in, even an `int32_t` variable
declaration becomes a parse error.

## Category 4, Duplicate symbols

**None real.** The two `object 'token' redefined` errors at
`autogenerated_cue_after.c:92` and `autogenerated_cue_before.c:92`
are likewise a parser-recovery artifact. With every preceding
declaration silently dropped due to syntax errors, CW8 reaches the
later `const css_token *token;` declaration and trips over what it
thinks is a re-declaration of an earlier one it half-parsed. It will
go away with the headers.

## Category 5, Other

The five `### Error: Compilation aborted at end of file ###` lines
are not real errors, they're CW8's bail marker after enough syntax
errors accumulate in a single TU. Five of them across 37 files means
five TUs hit the per-file error cap and got dropped; the rest emitted
errors all the way to EOF.

There are **zero** errors in this batch from:

- C99 features that survived our port audits
- Designated initializers (the Phase 4 conversions held)
- For-scope decls (Phase 4 hoists held)
- Forward enum decls (form_internal.h fix held)
- `inline` keywords (prefix file `#define inline` held)
- Mac toolbox / OT calls (not on the critical path of this batch)

This is significant: it means the libcss / NetSurf core porting work
from Phases 1-4 is **structurally correct**. The build is failing on
infrastructure (search paths), not on source-level C89 problems.

## Proposed fix order

Highest dependency first.

1. **[blocks all 2,084 errors]** Verify the Mac-side `MacSurf.mcp` is
   the renames-zip version. If not, re-unpack the renames-zip on the
   Mac, replacing the older project file. Re-attempt the build.

2. **[blocks all 2,084 errors if step 1 doesn't help]** Add the three
   new search-path entries as **absolute Mac-style paths** in the CW8
   IDE Project Settings → Access Paths → User Paths:
   - `…:browser:libcss:src:` (resolves `bytecode/*.h`,
     `parse/properties/*.h`)
   - `…:browser:netsurf:content:handlers:` (resolves `html/font.h`,
     `css/utils.h`, etc.)
   - `…:browser:netsurf:content:handlers:html:` (resolves the
     html-internal relative includes like `"private.h"`)
   The first two are mandatory; the third is needed by Phase 4
   html-handler files.

3. **[blocks 0 errors but cleanup]** Once the build passes, update
   `CLAUDE.md`'s "Access Paths (CodeWarrior)" list to include the
   three new entries so the next clean checkout has them documented.

4. **[blocks 0 errors but cleanup]** Decide whether to keep the
   Unix-style `..` paths in `MacSurf.mcp` (which work on Linux
   cross-check but are unreliable on Mac), or rewrite them as Mac
   absolute paths. Recommended: leave the .mcp file as-is for the
   Linux cross-check tooling, and treat the IDE access-path setting
   as the source of truth on the Mac side.

## What would break next (forecast)

When the headers resolve and these 37 files compile, the next batch
will hit files starting `g*` onwards. Likely follow-on issues based
on what we know is in the project but hasn't been exercised yet:

- **More missing-header cascades** in other html-handler files that
  include `"private.h"`, `"box.h"`, `"layout.h"` etc. relative to
  `content/handlers/html/`, these need search-path #12c (the third
  new entry above). Same fix.
- **Possible multi-arg `NSLOG` parse errors.** The macsurf prefix
  defines `NSLOG(catname, level, logmsg)` as a 3-arg macro, but
  ~hundreds of upstream call sites pass 4+ arguments. CW8 may or
  may not silently accept extras, if not, every multi-arg `NSLOG`
  is a separate error. This was flagged in Phase 4 but punted as
  "left as-is, the cross-check uses upstream variadic NSLOG."
- **`%zu` printf format** in `layout_flex.c` lines 722 and 914.
  CW8 MSL printf does not support `%zu`. Same Phase 4 punt.
- **The 29 renamed `ag_*` files** vs. the Mac-side stale references
  to the old `autogenerated_*` names, once the project file is
  fully resynced this is resolved, but during the transition the
  Mac may end up with both old and new files in the folder, and
  CW8 may pick up whichever it sees first.
